Lộ trình QA Engineer

Tháng 1 — Nền tảng & Tư duy QA ⏱ 3h Không hiểu sản phẩm thì không thể test đúng. QA cần đối chiếu tính năng với giá trị nghiệp vụ.

Chương 2: Nền tảng cơ bản về phần mềm

Yêu cầu: Chương Đã từng dùng ứng dụng web hoặc mobile

ELI5 — Tóm tắt 60 giây

Phần mềm giống một nhà hàng: menu là giao diện, bếp là backend, kho nguyên liệu là database, QA là người đảm bảo món ăn ra đúng và ngon.

Mục tiêu học

  • Hiểu thành phần cơ bản của một sản phẩm phần mềm
  • Phân biệt tính năng, dữ liệu và logic xử lý
  • Đặt câu hỏi đúng để hiểu nghiệp vụ

Nội dung chính

Phần mềm không chỉ là màn hình đẹp

Hành trình · Chương 2

Phần mềm không phải màn hình — là **input → xử lý → output → state**. Hiểu điều này, bạn test ít mà sâu hơn.

👉 Quick Challenge

UI báo 'Lưu thành công' nhưng refresh mất data. Bug ở đâu — UI hay backend?

✅ Gợi ý: Chưa kết luận — mở Network: API trả 200 nhưng DB không có row → backend/transaction. API 500 → backend. API 200 + DB OK → FE render sai state.
🧠 Think Like a QA

Với feature bạn test hôm nay: state nào thay đổi sau mỗi bước user?

Trước khi test, bạn cần hiểu phần mềm là gì — theo cách một QA cần hiểu, không phải theo cách lập trình viên viết code.

Một sản phẩm phần mềm thường gồm:

  • feature — tính năng người dùng nhìn thấy và dùng được (đăng nhập, đặt hàng, xuất báo cáo...).
  • requirement — mô tả cần làm gì và vì sao; là căn cứ để viết test và tranh luận "đúng/sai".
  • workflow — chuỗi bước nghiệp vụ: A → B → C (ví dụ: chọn sản phẩm → thanh toán → nhận email xác nhận).

Mỗi thao tác trong workflow có input (dữ liệu người dùng nhập hoặc hệ thống nhận) và output (kết quả hiển thị, file tải về, email gửi đi). Giữa các bước, hệ thống có state — trạng thái dữ liệu (giỏ hàng trống / có 2 món / đã thanh toán).

QA hay mắc lỗi chỉ nhìn output trên màn hình mà không hỏi: "Dữ liệu bên trong có đúng state không?" Chương 4 sẽ đi sâu frontend/backend; chương này xây nền tảng: mọi thứ bạn test đều là input → xử lý → output → state.

Cách QA "đọc" một tính năng mới

Khi nhận tính năng mới, đừng mở app và bấm ngay. Hãy trả lời 5 câu hỏi:

  1. Ai dùng? (vai trò: admin, khách vãng lai, nhân viên kho...)
  2. Mục tiêu nghiệp vụ? (tăng doanh thu, giảm sai sót, tuân thủ quy định...)
  3. Workflow chính? (happy path — đường đi lý tưởng)
  4. Input bắt buộc vs tùy chọn? (trường nào bỏ trống được, format gì)
  5. State thay đổi thế nào? (trước/sau thao tác, dữ liệu lưu ở đâu)

Ví dụ tính năng "Đặt lịch khám":

  • Input: ngày, giờ, bác sĩ, thông tin bệnh nhân.
  • Output: màn hình xác nhận + SMS/email.
  • State: slot lịch chuyển từ "trống" → "đã đặt"; nếu UI báo thành công nhưng state vẫn "trống" → đó là bug nghiêm trọng dù giao diện đẹp.

Thực hành ngay: chọn 1 màn hình trên DemoQA hoặc app bạn hay dùng, điền bảng Input | Output | State trước/sau cho một thao tác duy nhất.

⚠️ Common Trap

"Chỉ kiểm tra thông báo trên màn hình" — UI có thể 'nói dối' — backend fail nhưng toast vẫn hiện success.

🔼 Level Up

Mid QA luôn vẽ workflow + state diagram trước khi viết test case đầu tiên.

🪓 Real-world Trade-off — Story mơ hồ

Bắt đầu test ngay hay hỏi PO?

✅ Hướng xử lý: 15 phút hỏi làm rõ AC — rẻ hơn 2 ngày test sai hướng.

Requirement vs implementation — đừng nhầm

requirement trả lời "hệ thống phải làm gì". Implementation là cách dev đã code. QA test theo requirement; khi phát hiện lệch, bạn báo bug hoặc làm rõ với PO — không tự suy diễn "dev làm vậy nên chắc đúng".

Dấu hiệu requirement mơ hồ:

  • Story viết "User có thể quản lý đơn hàng" — không nói tạo/sửa/xóa/xem chi tiết.
  • Không có acceptance criteria — mỗi người hiểu một kiểu.
  • Không đề cập edge case: mạng chậm, nhập ký tự đặc biệt, quay lại bước trước.

Cách QA phản hồi chuyên nghiệp: "Em cần làm rõ: nếu user bỏ giữa workflow, dữ liệu có lưu nháp không? Acceptance criteria em đề xuất: Given..., When..., Then..."

Ba bẫy thường gặp ở chương này:

  • Chỉ test UI, không kiểm tra luồng dữ liệu và state.
  • Test từng màn hình rời, không test cả workflow.
  • Nhầm bug implementation với thay đổi requirement chưa được PO duyệt.

Kế hoạch rèn luyện 7 ngày — Tuần nền tảng phần mềm

Lịch 30–45 phút/ngày, tập trung hiểu sản phẩm thay vì tìm lỗi ngẫu nhiên:

  • Ngày 1: Chọn 1 app, liệt kê 5 feature chính và ai dùng mỗi feature.
  • Ngày 2: Vẽ workflow đăng ký hoặc đặt hàng bằng text (mũi tên A → B → C).
  • Ngày 3: Với 3 màn hình quan trọng, ghi bảng Input | Output | State trước/sau.
  • Ngày 4: Mỗi workflow tìm 2 edge case (bỏ trống trường bắt buộc, bấm Back giữa chừng...).
  • Ngày 5: Chọn 1 requirement mơ hồ (tự đặt giả định), viết 3 câu hỏi làm rõ gửi "PO".
  • Ngày 6: Viết 1 user story + 2 acceptance criteria cho feature đơn giản (ví dụ: đổi mật khẩu).
  • Ngày 7: Tổng kết "Bộ câu hỏi chuẩn khi nhận feature mới" — 5–7 câu in ra dán bàn làm việc.

Sau tuần này bạn sẽ không còn test kiểu "bấm lung tung". Chương 3 tiếp theo đặt QA vào SDLC — vòng đời phát triển phần mềm và khi nào bạn nên tham gia test.

Checkpoint — Tự kiểm tra nhanh

Câu 1: 🧩 Quick Challenge: UI báo 'Lưu thành công' nhưng refresh mất data. Bug ở đâu — UI hay backend?

✅ Chưa kết luận — mở Network: API trả 200 nhưng DB không có row → backend/transaction. API 500 → backend. API 200 + DB OK → FE render sai state.

Câu 2: 🧠 Think Like a QA: Với feature bạn test hôm nay: state nào thay đổi sau mỗi bước user?

Ghi câu trả lời 3–5 dòng hoặc Expected/Actual có evidence trên app thật.

Câu 3: ⚠️ Common Trap: "Chỉ kiểm tra thông báo trên màn hình" — bạn tránh được chưa?

UI có thể 'nói dối' — backend fail nhưng toast vẫn hiện success.

Ví dụ làm mẫu

Issue note — Expected vs Actual

Vấn đề: Nút Submit không phản hồi sau khi điền form.

Bước: (1) Mở form (2) Điền email hợp lệ (3) Click Submit

Expected: Thông báo thành công hoặc lỗi validation rõ.

Actual: Nút xám 3 giây, không thông báo.

Hỏi ai: Dev — Bug report đầy đủ: Ch8.

Nên
  • Hỏi yêu cầu nghiệp vụ trước khi viết test
  • Theo dõi state trước và sau thao tác
  • Xác minh kết quả bằng dữ liệu thực
Không nên
  • Chỉ check thông báo trên UI
  • Bỏ qua edge case của workflow
  • Giả định implementation lúc nào cũng đúng

Ví dụ thực tế

📦 Mini Scenario: Tính năng đặt lịch

Tình huống: Người dùng đặt lịch thành công trên UI nhưng backend không lưu state. QA chỉ ra sự khác nhau giữa thông báo hiển thị và dữ liệu thực.

👉 Quick Challenge

Nếu bạn là QA trong tình huống này — quyết định đầu tiên là gì?

✅ Gợi ý: Quantify rủi ro + đề xuất hành động cụ thể (test thêm / rollout / delay) — không chỉ mô tả sự việc.

Vì sao cách này hiệu quả: Stakeholder cần data để quyết định, không cần drama. QA tạo giá trị bằng clarity.

Trade-off: Tốc độ release vs coverage — luôn ghi rõ phần chưa test và workaround nếu ship.
